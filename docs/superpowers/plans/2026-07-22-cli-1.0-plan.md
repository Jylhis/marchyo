# Marchyo CLI 1.0 — Implementation Plan

Date: 2026-07-22
Spec: [2026-07-22-cli-1.0-design.md](../specs/2026-07-22-cli-1.0-design.md) (merged in #127)
Baseline: marchyo-cli 0.1.0, main @ `2ebb628`

Ten PRs across seven phases. Each PR is independently green (`just fmt`,
`just check`, `bun test` in `packages/marchyo-cli`) and leaves every
keybinding working. Phases 3–6 all depend only on Phase 1 and may land in
any order; everything else is sequential.

Conventions for every PR below:

- New options: declaration-only, platform-neutral files under
  `modules/nixos/options/` (darwin eval gate).
- Every port PR that absorbs a script: flip the binds/menu entries it owns
  to `marchyo …`, delete the `writeShellApplication`, extend the VM
  checklist in the PR description, and verify live via `just run`.
- CLI code: commands in
  `packages/marchyo-cli/packages/user-cli/src/commands/`, shared logic in
  `packages/core/src/`, unit tests with the existing mocked-exec pattern.
- Never auto-`nixos-rebuild` in CI; system-touching commands get
  `--dry-run` coverage only.

## Phase 0 — Safety net (PR 1: CI + maintenance recipes)

Independent of everything; land first.

1. Add `.github/workflows/cli.yml`: triggers on PRs/pushes touching
   `packages/marchyo-cli/**` or the workflow file itself; single
   ubuntu-latest job with `oven-sh/setup-bun@v2`, then
   `bun install --frozen-lockfile` → `bun run typecheck` → `bun test`
   (mirror `site.yml` structure, `timeout-minutes`, concurrency group).
2. Add `just cli-repin` recipe: rebuild
   `.#legacyPackages.x86_64-linux.marchyo-cli`, extract the suggested FOD
   hash, patch `outputHash` in `packages/marchyo-cli/package.nix`.
3. Verification: the PR itself runs the new workflow; `just cli-repin`
   is a no-op when the lockfile is unchanged.

## Phase 1 — F3.0 engine (PR 2: core modules + `runtime` command)

The foundation every mutating command consumes.

1. `packages/core/src/runtime-state.ts`
   - Schema: `{ schemaVersion: 1, overrides: Record<string, JsonValue> }`.
   - Path: `$XDG_STATE_HOME/marchyo/runtime.json`
     (fallback `~/.local/state/marchyo/runtime.json`).
   - API: `loadRuntimeState`, `saveRuntimeState` (write temp file in the
     same directory + rename), `setOverride`, `clearOverride`,
     `listOverrides`. Unknown/newer `schemaVersion` → treat as empty +
     warn on stderr (files are internal; never hard-fail on them).
2. `packages/core/src/desktop.ts`
   - Typed actuator wrappers over the existing `system.ts` runner:
     `hyprctl` (keyword get/set, `reload`, `dispatch`), `makoctl`,
     `systemctlUser`, `swayosdClient`, `notify` (notify-send).
   - All failures surface as ordinary command errors (exit 1) with the
     existing missing-binary handling from `system.ts`.
3. `packages/core/src/apply.ts`
   - `type ChangeSpec = { key; runtimeApply(rt); runtimeRevert(rt);
     stateWrite(value); stateDelete() }`.
   - `applyChange(rt, spec, { mode: "runtime"|"apply"|"revert", value })`:
     - runtime: `runtimeApply` + `setOverride`.
     - apply: runtime path + write `cli-state.json` (`state.ts`) +
       `nixos-rebuild` (`nix.ts`/`flake.ts`), then `clearOverride` (the
       declarative value now matches).
     - revert: `clearOverride` + `runtimeRevert`; if the key is persisted
       in `cli-state.json`, also delete it + rebuild.
   - `parseChangeFlags` commander helper: `--apply` and `--revert` are
     mutually exclusive → usage error (exit 2).
4. New command `marchyo runtime` (`status` | `restore`):
   - `status`: list active overrides (text table / `--format json`).
   - `restore`: re-run `runtimeApply` for each stored override;
     idempotent, silently no-op when none. Restore resolution: each
     command group registers its ChangeSpecs in a central registry
     (`user-cli/src/changes.ts`) keyed by dotted key, so `restore` can
     rehydrate without duplicating logic.
5. HM wiring: in `modules/home/hyprland.nix` (gated on
   `osConfig.marchyo.cli.enable or false`), add `exec-once = marchyo
   runtime restore` and include it in the post-`hyprctl reload` hook used
   by theme-runtime.
6. Tests: full mode-matrix unit tests (runtime/apply/revert × override
   present/absent × persisted/not), runtime-state corruption cases, and
   an eval test asserting the exec-once lands when `cli.enable` is on.

## Phase 2 — Theme slice: engine proof + F3.3

### PR 3: N-theme build layer (Nix only, no CLI change)

1. Option `marchyo.theme.themes` (`types.listOf types.str`, default
   `[ "jylhis-dark" "jylhis-light" ]`) in `modules/nixos/options/theme.nix`.
2. Generalize `modules/home/theme-runtime.nix` from the two-variant #118
   layout to N entries:
   - `jylhis-dark`/`jylhis-light` build from design tokens as today; any
     other name resolves a `pkgs.base16-schemes` YAML through the
     existing `marchyo.theme.scheme` machinery.
   - Per-theme asset dir (waybar CSS, ghostty, mako, GTK, fzf, bat,
     starship, hyprlock colors) + a build-generated
     `~/.local/share/marchyo/themes/manifest.json`:
     `[{ name, variant, dir }]` where `variant` (dark|light) is computed
     from base00 luminance for base16 themes.
   - Assertion: unknown theme name fails eval with the list of valid
     scheme names.
3. Eval tests: default pair, a 4-theme list incl. two base16 names, the
   unknown-name assertion. Verify closure impact stays proportional to
   list length (spot-check `nix path-info -S` locally, not in CI).

### PR 4: CLI theme commands (first ChangeSpec consumer)

1. Absorb `marchyo-theme-toggle` (`theme-runtime.nix`) into
   `commands/theme.ts` as the theme ChangeSpec: `runtimeApply` = pointer
   symlink swap + consumer reloads; `stateWrite` = theme selection into
   `cli-state.json`.
2. Commands: `theme list` (manifest), `theme get`, `theme set <name>
   [--apply|--revert]`, `theme next` (manifest order, wraps).
3. Back-compat: `dark`/`light` alias to the Jylhis pair; `--rebuild`
   accepted as deprecated alias for `--apply` (stderr warning).
4. Flip callers: hyprland bind / menu Style entry / waybar hooks that
   invoke `marchyo-theme-toggle` now call `marchyo theme next` (or `set`);
   delete the script in the same PR.
5. `bg set <path>` / `bg next`: drive `awww img` (the mechanism
   `hyprland.nix:37` uses). `bg next` cycles the images shipped by the
   wallpaper package (`marchyo.theme.wallpaper.package`, default
   `marchyo-wallpapers`), current selection tracked as a runtime override.
   Note: today the wallpaper is one variant-keyed file; extending
   `marchyo-wallpapers` to a per-theme *set* is part of this task.
6. Tests: unit (mocked hyprctl/awww/manifest IO), eval tests for changed
   modules, VM checklist: switch among 3 themes live, `--apply` dry-run,
   `runtime restore` after `hyprctl reload` keeps the runtime theme.

## Phase 3 — Toggles slice (PR 5)

Depends on Phase 1 only (as do Phases 4–6).

1. `commands/toggle.ts` + per-toggle ChangeSpecs in `changes.ts`:
   `gaps`, `transparency`, `nightlight`, `waybar`, `touchpad`,
   `touchscreen`, `idle`, `screensaver`, `notifications`, `suspend`,
   `hybrid-gpu` (`--apply`-only; usage error on runtime mode).
   `toggle <name>` flips; `toggle <name> on|off` sets.
2. Absorb + delete: `marchyo-nightlight-toggle`, `marchyo-idle-toggle`,
   `marchyo-dnd-toggle`, `marchyo-screenrecord-toggle` stays for Phase 4
   (recording is capture, not a toggle), `marchyo-zoom` becomes
   `toggle zoom`? — **no**: zoom is a hyprctl one-shot, it moves to Phase 5
   as `marchyo zoom` alongside launch helpers. Sources:
   `window-toggles.nix`, `screensaver.nix` (idle/screensaver toggles),
   `waybar.nix` `dndStatus` (waybar `custom/dnd` exec + on-click →
   `marchyo toggle notifications` / a `--status` probe flag).
3. Flip binds in `hyprland.nix` / `omarchy-binds.nix` / waybar on-click
   handlers; delete absorbed scripts.
4. Tests: unit per toggle (both directions + revert), eval tests, VM
   checklist covering each bind.

## Phase 4 — Capture slice (PR 6)

1. `commands/capture.ts`: `screenshot [--edit]` (grimblast, satty when
   `--edit`), `record [--audio none|desktop|mic]` (absorbs
   `marchyo-screenrecord-toggle`), `ocr` (grimblast → tesseract →
   wl-copy), `color` (hyprpicker).
2. Today's screenshot binds are *inline* grimblast pipelines in
   `screenshot.nix` — replace each `bindd` exec with the matching
   `marchyo capture …` invocation (area/window/output/screen variants
   become flags: `--target area|active|output|screen`, extending the
   frozen surface before the freeze).
3. Tests: unit (mocked pipelines incl. audio device selection), eval,
   VM checklist per bind.

## Phase 5 — Menus, launch & power slice (PR 7)

1. `commands/menu.ts`: `menu` and `keybindings` keep the gum/floating
   ghostty *presentation* (absorbed from `menus.nix` `systemMenu` /
   `powerMenu` and `keybindings-cheatsheet.nix`), but every dispatched
   action becomes a `marchyo …` call.
2. `launch <app>` / `focus-or-launch <app>` (hyprctl dispatch +
   window-class probe); absorb `marchyo-file-manager-cwd`,
   `marchyo-monitor-scale-cycle`, `marchyo-laptop-display-toggle`
   (surface as `marchyo monitor scale-cycle` / `marchyo monitor
   laptop-toggle`), and `marchyo zoom` from Phase 3's deferral.
3. Power: `lock`, `logout`, `suspend`, `reboot`, `shutdown`,
   `powerprofile get|list|set` (powerprofilesctl).
4. Flip menu entries + binds; delete absorbed scripts.
5. Tests: unit, eval, VM checklist (menus open, every entry dispatches).

## Phase 6 — Utilities slice (PR 8)

1. `commands/transcode.ts` (`marchyo-transcode`), `commands/share.ts`
   (`marchyo-share` → `marchyo share <file>`; `nautilus.nix`'s
   LocalSend action calls it), `commands/reminder.ts`
   (`reminder set|show|clear` absorbing the three `marchyo-reminder-*`
   scripts and their transient systemd timers), quick-info notifies
   (`marchyo-notify-datetime|battery` → `marchyo info datetime|battery`).
   These are surface *additions* agreed during planning (pre-freeze, so
   allowed); the spec's command table gains `share`, `reminder`, `info`,
   `monitor`, `zoom` rows at freeze time.
2. `font list|current|set <name>` — runtime-first ChangeSpec over the
   ghostty/starship font wiring; `--apply` writes the font selection.
3. Flip callers in `utilities.nix`, `waybar.nix`, `nautilus.nix`,
   `omarchy-binds.nix`; delete absorbed scripts.
4. Tests: unit, eval, VM checklist.

## Phase 7 — Declarative group + freeze + release

### PR 9: declarative commands

1. `install <feature>` / `remove <feature>` / `toggle <feature> --apply`
   for the coarse flags (desktop/development/media/office + the
   per-feature namespaces): edit `cli-state.json`, rebuild; refuse and
   hint when the running flake doesn't wire `marchyoCliState` (probe: the
   current system's config lacks the sidecar marker — reuse the
   `options-eval.ts` machinery).
2. `webapp add <url> [--name --key]` / `webapp rm <name>` →
   `marchyo.webapps.apps` entries in `cli-state.json` + rebuild;
   key-collision check against the documented taken `SUPER+SHIFT` letters.
3. `security enroll fido2|fingerprint`: wrap `pamu2fcfg` /
   `fprintd-enroll`, requiring the matching `marchyo.security.*` flag
   (error with the option name when off).
4. Tests: unit (state-file mutations, collision/refusal paths, dry-run),
   eval.

### PR 10: contract freeze + docs + packaging + 1.0.0

1. Contract snapshot tests: `--help` trees of every command and
   `--format json` shapes of `status`, `theme get/list`, `toggle`,
   `runtime status`, `debug` — snapshots checked in; CI fails on drift.
2. `marchyo completion bash|zsh|fish`; `package.nix` generates + installs
   completions and a man page from the command tree at build.
3. Docs: `site/src/content/docs/docs/usage/cli.mdx` (full reference,
   `marchyoCliState` opt-in, `--impure` story); regenerate `hotkeys.mdx`
   (binds now show `marchyo …`); `updating.mdx` gains
   `marchyo update/upgrade/rollback`; sync `packages/marchyo-cli/README.md`
   and `CLAUDE.md`/`AGENTS.md` option references.
4. Version 1.0.0: single-source from `package.json` (`version.ts` and
   `package.nix` read it); update the deprecation copy for `--rebuild`.
5. Gate re-check against the six release criteria in the spec, then tag
   `marchyo-cli-v1.0.0`.

## Risks & mitigations

- **Engine API churn** (biggest risk): PR4 is deliberately the only
  consumer before fan-out — treat it as the API review; adjust
  `ChangeSpec` there, not in five parallel PRs.
- **Bind regressions**: every port PR carries a VM checklist; hotkeys
  docs regenerate only once at PR10 to avoid churn.
- **FOD hash drift**: every PR touching `bun.lock` runs `just cli-repin`;
  CI's nix build catches misses.
- **Wallpaper set scope creep** (PR4 item 5): if per-theme wallpaper sets
  balloon, ship `bg set` only and defer `bg next` to a follow-up — it is
  not one of the six release criteria.
- **Surface additions** (`share`, `reminder`, `info`, `monitor`, `zoom`,
  capture `--target`): recorded here, folded into the frozen table at
  PR10; anything not ready by then ships post-1.0 instead of blocking.
